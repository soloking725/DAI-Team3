1. What is needed for the MVP that a typical LLM can code for you

    (These are pure code-generation tasks. Paste these prompts into any capable LLM, and it will write 90% of the boilerplate correctly.)

    Database Migration (persistence.py → PostgreSQL/Supabase)
    Prompt: "Rewrite my persistence.py using SQLAlchemy (async) to connect to Supabase PostgreSQL. Keep the exact same API (get_vera_state, save_session, etc.) but replace JSON file I/O with DB queries. Add a college_id foreign key and a student_id primary key. Write the table schema."

    Basic Authentication Scaffold (Google SSO / Supabase Auth)
    Prompt: "Implement a login page using st.button('Login with Google') that redirects to Supabase OAuth. On callback, store the user's id, email, and college_id in st.session_state. Redirect to the timeline if logged in, else show login."

    DSO Student List Dashboard (Read-Only + Manual Edit)
    Prompt: "Create a new Streamlit page (pages/20_DSO_Dashboard.py). Query the database for all students in the current college. Display them in st.dataframe with columns: Name, Visa Type, Current Step (dropdown to manually change it), and a 'View Timeline' button. Add a simple filter by step status."

    Static School Guide Upload (Link-only)
    Prompt: "Add an upload widget on the DSO dashboard that uploads a PDF to an S3/R2 bucket and stores the public URL in the colleges table under guide_pdf_url. Show a download button on the student timeline page pointing to this URL."

    Linter / Formatter Setup
    Prompt: "Generate a .pre-commit-config.yaml and pyproject.toml with ruff and black configuration for this Streamlit project."

2. What is needed for the MVP that YOU (or a human engineer) should handle manually

    (These are deployment, configuration, and architectural decisions. LLMs cannot click UI buttons or debug network issues for you.)

    Supabase / PostgreSQL Project Setup
    Task: Create a Supabase account, create a new project, run the SQL schema (generated by the LLM) in the Supabase SQL Editor, and set up Row-Level Security (RLS) policies for college_id.

    OAuth App Registration
    Task: Go to Google Cloud Console, create a new OAuth 2.0 Client ID, configure the redirect URI to your Streamlit Cloud URL, and copy the Client ID/Secret into Streamlit Cloud secrets (st.secrets).

    S3 / R2 Bucket Configuration
    Task: Create a Cloudflare R2 bucket (or AWS S3), generate API keys with write permissions, set CORS policy to allow uploads from your Streamlit Cloud domain, and store the keys in st.secrets.

    ChromaDB Re-ingestion (If adding college_id to metadata)
    Task: Run a one-off local Python script (generated by the LLM) to wipe the existing ChromaDB collection and re-ingest all documents with college_id = "pilot_school_id". This cannot be done via the browser.

    Testing the Auth Flow
    Task: Actually log in with a test Google account, debug the redirect loop, fix CORS issues, and ensure the JWT is correctly parsed.

3. Design Choices We Need to Make (UI/UX)

    (These are conceptual decisions that determine what the LLM actually builds.)

    Single Pilot College vs. Multi-Tenant Signup

        Choice for MVP: Hardcode one college (e.g., college_id = 1). Do NOT build a "College Registration" wizard. Let the DSO set up the guide via a secret admin page (/admin_setup) accessible only to a hardcoded email.

    Student Onboarding Flow

        Choice: Students log in with Google SSO. If their email domain matches the pilot school, they are automatically assigned to that college_id. If not, show "Not authorized for this pilot." No self-serve signup forms.

    Timeline Interaction for DSOs

        Choice: DSOs can overwrite a student's step status via a dropdown in the dashboard. Do NOT build a "Student vs. DSO status" conflict UI yet—just let the DSO override it (with a note: "Manually updated by DSO").

    Chatbot Placement

        Choice: Keep the chatbot as a floating popover (shared/components.py -> render_floating_chat()) on the timeline page for students. For DSOs, hide the chatbot entirely—they only see the dashboard.

    Mobile Support

        Choice: Desktop-first. The DSO dashboard is unusable on mobile anyway. Students can use the timeline on mobile, but it will be a single-column stack (accept the slight ugliness).

    PDF Extraction

        Choice: Do NOT parse PDFs in the MVP. Just host the file as a download link. Tell the pilot school: "Upload your guide here, your students will see a download button." This sidesteps the 60-second timeout entirely.

4. What is needed for the future, with issues explained in depth

    (These features are essential for scaling beyond the pilot school, but they require infrastructure that Streamlit Cloud cannot provide.)

A. Background Workers (Cron / Scheduling)

    The Feature: Daily consulate scraping, automated "Did you complete this step?" email reminders, and cohort report generation.

    The Problem: Streamlit Cloud has no persistent background processes. A script that runs longer than 60 seconds or needs to execute at 3 AM will fail.

    The Solution: Move these tasks to a separate service (e.g., Render Cron Jobs, AWS Lambda + EventBridge, or Supabase Edge Functions). Your Streamlit app just reads the cached results from the DB.

B. Real-Time Communication & Notifications

    The Feature: DSOs sending messages to students, and students receiving push notifications/email alerts when their status changes.

    The Problem: Streamlit does not support WebSockets. You cannot push a message to a DSO's dashboard the moment a student submits a question.

    The Solution: Use Pusher, Supabase Realtime, or Ably for WebSockets, or fallback to email via SendGrid/Resend. This requires a separate microservice or a third-party integration.

C. Bulk Operations (CSV Import/Export & Batch Status Updates)

    The Feature: DSOs importing 200 students via CSV and exporting final reports at semester end.

    The Problem: Streamlit's st.dataframe does not handle row selection well across reruns, and processing a 200-row CSV in a single request can hit memory/time limits.

    The Solution: Move this logic to a FastAPI backend running on a VM (Render/Fly.io). Streamlit just calls the API endpoint POST /api/import_roster which processes the file asynchronously and returns a job_id for status polling.

D. Multi-Tenant Configuration & Billing

    The Feature: Each college has its own branded login, custom steps, and usage limits (e.g., 1,000 LLM requests/month).

    The Problem: config.py uses a single .env file. You cannot have per-college API keys, rate limits, or billing caps without a multi-tenant database design.

    The Solution: Store api_key, rate_limit, and branding in the colleges table. Enforce rate limits using a Redis counter keyed by college_id (Redis is a separate hosted service like Upstash).

E. Audit Logs & Compliance (FERPA/GDPR)

    The Feature: Tracking exactly who viewed which student's timeline and providing a "Right to Erasure" delete path.

    The Problem: Your current JSON files have no audit_logs table or deleted_at soft-delete column.

    The Solution: Add an audit_logs table with admin_id, student_id, action, and timestamp to the database schema. For deletions, implement a soft-delete (mark deleted_at) and a 30-day retention policy before hard-deleting.

F. Prompt Injection & Data Exfiltration Defenses

    The Feature: Preventing a malicious student from asking Vera to reveal other students' PII or bypass fees.

    The Problem: Your safeguards (classify_input, filter_output) run inside Streamlit (the frontend). A malicious user could theoretically intercept the API call and disable them.

    The Solution: Move all LLM safeguards to the backend API. The Streamlit frontend sends the user message to POST /api/chat, and the backend validates permissions, applies filter_output, and returns the sanitized response. Never trust the frontend.

G. Migration from ChromaDB to a Scalable Vector DB

    The Feature: Adding college_id to every vector query so that College A never retrieves College B's uploaded docs or scraped consulate pages.

    The Problem: ChromaDB does not support schema migrations. Adding a new metadata field requires re-ingesting all documents from scratch.

    The Solution: Implement a versioned ingestion pipeline (e.g., collection_v2). On migration, run a background job to rebuild the new collection while the old one remains live, then swap the collection name in config.py with zero downtime.

H. Token Cost Tracking & Auto-Cutoff

    The Feature: Preventing a single college from using 1 million tokens and racking up a $500 bill in a day.

    The Problem: No middleware tracks token usage per college_id.

    The Solution: Implement a token counter in the backend API that increments a monthly_token_usage column in the colleges table. If monthly_usage > quota, return a 429 (Too Many Requests) error instead of calling the HPC.

I. Data Portability (SIS Integration)

    The Feature: Colleges exporting student visa statuses back into their internal SIS (Banner, Workday).

    The Problem: No REST API exists for external systems to pull data.

    The Solution: Build a dedicated FastAPI REST layer (GET /api/students/export?cohort=fall2025) that returns JSON/CSV, secured by an API key stored in the college's profile. This is a purely backend feature—Streamlit is just the UI.

J. Dynamic Per-Student / Per-Cohort Steps

    The Feature: DSOs adding custom steps (e.g., "Submit health form by 05/01/2026") to a specific cohort without affecting other cohorts.

    The Problem: timeline_data.py is a static global dictionary. You cannot have one timeline for Fall 2025 and another for Spring 2026.

    The Solution: Store timeline definitions in the database (cohort_templates table). When a student registers, clone the template steps into a student_steps table (so changes to the template don't retroactively change existing students). This requires complex DB modeling that goes far beyond JSON files.
